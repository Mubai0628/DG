# v0.12 Git / Shell Safe Lanes Post-Release Review

This review locks the v0.12 Git / Shell Safe Lanes MVP release line. P0P is
complete and should not expand into arbitrary Git, arbitrary shell, Git write
commands, install/network/destructive commands, native bridge, desktop action,
or broad capability execution.

The next phase is P0Q: App Live Proposal Generation MVP. P0Q may add an
explicit user-confirmed App path for live DeepSeek patch proposal generation,
but it must not auto-apply, write files from model output, rollback from model
output, execute Git or shell, issue broad PermissionLeases, or bypass the
existing schema / repair / validation / audit / approval / approved apply /
verification / rollback / replay chain.

## Current Capabilities

- `web_table_to_csv` Convert remains the real conversion flow.
- Event Log / Replay remains available.
- Record Draft Event remains the App/Tauri local summary-event write path.
- App approved apply / rollback remains available only through human approval,
  typed confirmation, path/content gates, private checkpoint metadata, and
  summary-only events.
- Summary-only apply / rollback events remain replayable.
- Git read lanes are available for status summary, diff summary, log summary,
  and branch summary only.
- Shell verification allowlist lanes are available through fixed templates only.
- Verification Surface shows summary-only Git and shell lane results.
- Verification summary events are summary-only and replayable.
- Verification replay / control projection can reference verification evidence.
- Approved execution + verification smoke proves approved apply, verification
  lanes, rollback, and replay remain connected without raw output.

## Disabled Capabilities

- DeepSeek auto-apply.

## Boundary Lock

P0P proves verification can be attached to approved apply / rollback only through
fixed Git read lanes, fixed shell verification templates, summary-only
verification events, and replay/control projections.

P0Q must preserve that boundary. A live model may produce a structured patch
proposal only after explicit user confirmation and opt-in policy checks. The
proposal must then enter the existing schema, repair, validation, audit,
approval, approved execution, verification, rollback, and replay chain before
any user workspace mutation is possible.

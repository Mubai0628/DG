# v0.13 App Live Proposal Generation Post-Release Review

This review locks the v0.13 App Live Proposal Generation MVP release line. P0Q
is complete and should not expand into auto-apply, autonomous coding loops,
broad PermissionLease issuance, arbitrary Git or shell execution, native bridge,
desktop action, or raw model payload persistence.

The next phase is P0R: End-to-End Coding Task MVP. P0R may connect the existing
live proposal, approved apply, verification, rollback, event, and replay
surfaces into one reliable coding task flow, but every mutation must remain
human-approved, typed-confirmed, checkpointed, summary-only, and replayable.

## Current Capabilities

- `web_table_to_csv` Convert remains the real conversion flow.
- Event Log / Replay remains available.
- Record Draft Event remains the App/Tauri local summary-event write path.
- App live proposal generation is available through explicit user opt-in only.
- Live proposal output enters repair, schema validation, model proposal import,
  and chain integration before it can be considered for any approved execution
  path.
- Proposal repair/schema/import/chain previews remain summary-only.
- Approved apply and rollback remain available only through the existing human
  approval receipt, exact typed confirmation, path/content safety checks,
  checkpoint metadata, summary-only events, and replay visibility.
- Git and shell safe lanes remain fixed, bounded, summary-only verification
  lanes after approved execution.

## Boundary Lock

P0Q proves the App can request one live DeepSeek patch proposal only after
explicit opt-in, session receipt, exact typed confirmation, API key source-ref
policy, request-boundary, redaction, repair, schema, import, and chain gates.

P0R must preserve that boundary while connecting the already-built pieces into
one end-to-end coding task flow:

```text
User objective
-> live DeepSeek proposal generation
-> repair / schema validation
-> model proposal import
-> chain integration
-> diff / validation / approval
-> human typed confirmation
-> approved apply
-> Git / shell verification safe lanes
-> summary events / replay
-> rollback if needed
```

P0R must not introduce auto-apply, autonomous coding, broad PermissionLease,
arbitrary Git/shell, native bridge, desktop action, raw model payload events, or
raw workspace payload events.

# v0.2 Control Plane Roadmap

## Current Status

DeepSeek Workbench v0.1 has completed the first safe vertical slice:

- browser extension visible-table capture with sanitized `BrowserDomPayload`.
- local web-table-to-CSV runtime flow.
- `ToolBroker` and guarded `fs.write_draft`.
- workspace drafts under `workspace/drafts/`.
- event log and replay summary.
- Tauri desktop shell RC.
- Event Log / Replay desktop panel.
- bridge threat model, protocol dry harness, preview gate, Native Messaging
  manifest dry check, and fake native host harness.

## Route Adjustment

P0E real transport work is paused after the fake native host harness. Do not
continue into P0E-008 yet. The next stage moves from capability stacking into a
v0.2 control plane.

The project is now positioned as a DeepSeek-native local coding workbench. The
control plane should make model capability, context assembly, permissions,
memory policy, agent state, and execution side effects explicit before any new
automation surface is added.

## Explicit Non-Goals

v0.2 planning does not authorize:

- real Native Messaging implementation.
- localhost bridge.
- browser extension auto-send.
- automatic Convert.
- desktop operator or desktop action.
- dynamic agent bidding.
- mutating MCP, plugins, or skills.
- memory system writes.
- patch apply.
- automatic file writes outside approved tool execution.

## Control Plane Modules

### Model Plane

The Model Plane is the source of truth for DeepSeek capability profiles. It
tracks model IDs, legacy aliases, thinking behavior, tool-call compatibility,
JSON output, cache usage fields, reasoning content, and probe invariants.

### Context Plane

The Context Plane owns prompt assembly, context ledger policy, cache boundary
events, context provenance, and injection-aware text handling.

### Capability Plane

The Capability Plane maps user-visible capabilities to explicit tools and
permissions. It must describe whether a capability is read-only, proposal-only,
or execution-capable.

### Memory Plane

The Memory Plane is not implemented in this step. Future work must define
normal/reject/injection/event tests before any memory write path exists.

### Agent Plane

The Agent Plane will coordinate task state and model/tool turn policy. It must
not introduce dynamic agent bidding before the permission and event model is
settled.

### Execution Plane

The Execution Plane owns side effects. Every side effect must follow:

```text
propose -> validate -> approve/reject -> execute -> result
```

The existing web-table-to-CSV runner remains the reference example.

## Phase Plan

### P0F: Control Plane Foundation

- Model Plane capability profile and dry probe.
- rules-and-ledger upgrade.
- control-plane event taxonomy.
- proposal/approval vocabulary shared by runtime and desktop.

### P0G: Context and Capability Planes

- context provenance and budget policy.
- capability registry.
- injection-aware capability tests.
- read/propose/execute permission model.

### P0H: Agent and Execution Planes

- deterministic local agent state machine.
- side-effect proposals.
- approval and rejection events.
- execution results with replay support.

### P0I: Memory Plane Design Gate

- memory threat model.
- memory schemas.
- no write path until normal/reject/injection/event tests exist.
- retention and deletion policy.

## Done Standards

Each control-plane module must include:

- normal-path tests.
- reject-path tests.
- injection or adversarial-input tests.
- event/replay tests where state changes are recorded.

Every side effect must remain gated by proposal, validation, approval or
rejection, execution, and result recording.

The existing gates must not regress:

- `pnpm verify:ci`
- `pnpm release:smoke`
- `pnpm app:qa:check`
- conformance dry passes.
- conformance live remains opt-in and skipped by default.
- boundary and secret checks remain green.

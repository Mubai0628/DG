任务目标：完成所有小阶段任务。

```text

```

```text
P1H — MCP / Plugin / Skill Controlled Execution Hardening
```

本阶段不是打开新的大能力，而是对已经进入体系的外部能力做安全加固：

- MCP read-only tool execution consistency
- Plugin / Skill sandbox escape checks
- Capability Broker enforcement hardening

- arbitrary MCP tool invocation
- arbitrary plugin code execution
- arbitrary skill runtime execution

- broad native bridge
- broad desktop automation

- arbitrary Git / shell
- broad PermissionLease
- raw tool output / raw args / secrets in events

## 0. 全局工作方式

每个普通小阶段必须：

1. 开始前检查 Git 状态。
2. 只修改本小阶段相关文件。
3. 运行 scoped checks。
4. 本地 commit。
5. 不 push。
6. 不 tag。
7. 不创建 GitHub Release。

最终 RC 小阶段才允许：

1. 跑 full gates。
2. push main。
3. 等 GitHub Actions green。
4. tag。
5. push tag。
6. 创建 GitHub pre-release。

如果遇到阻塞：

- 停止。
- 不要绕过安全边界。
- 不要创建 half release。
- 给出 blocker report。

---

## 1. 全局 Git workflow

每个小阶段开始时运行：

```powershell
git status --short
git status -sb
git log --oneline origin/main..HEAD
```

如果存在与当前小阶段无关的 dirty changes，停止并报告。

每个小阶段结束时运行：

```powershell
git diff --check
git diff --cached --check
```

然后只 stage 当前小阶段文件并 commit。

普通小阶段禁止：

```text
git push
git tag
gh release create
```

最终 release 小阶段才允许 push/tag/release。

---

## 2. 全局测试策略

普通小阶段只跑 scoped checks。

禁止普通小阶段运行：

```text
pnpm test                       # 无过滤全量
pnpm verify:ci
pnpm release:smoke
pnpm app:build
browser-extension build/test
eval:web-table-to-csv
full conformance matrix
```

Runtime tests 必须优先使用显式 Vitest 文件路径，例如：

```powershell
pnpm exec vitest run runtime/test/<specific>.test.ts
```

阶段收口 / release 小阶段必须跑 full gates。

---

## 3. 全局硬边界

整个 v0.30 阶段禁止：

- mutating MCP tools
- arbitrary MCP tool invocation
- arbitrary plugin code execution
- arbitrary skill runtime execution
- plugin installation with code execution
- skill runtime with filesystem/network/process access
- native bridge broad action
- broad desktop automation
- autonomous arbitrary tool execution
- arbitrary Git / shell
- broad PermissionLease
- auto-apply
- model-driven file write
- EventStore events with raw tool args/output
- raw tool output in App surfaces
- raw plugin/skill package content in events
- API key / Authorization / Bearer / token persistence
- process spawn / child_process / exec / shell
- file system mutation outside already-approved workspace apply/rollback lanes
- expanding desktop actions beyond current approved lanes

Allowed existing capabilities must remain within current boundaries:

- App-side approved apply / rollback remains human-approved and rollbackable.
- Git/shell verification safe lanes remain fixed and summary-only.
- MCP read-only tool execution remains fixed-profile, allowlisted, approved, bounded, and summary-only.
- Plugin / Skill remains manifest / metadata / safe simulation only unless explicitly covered by no-arbitrary-execution contract.
- Capability Broker remains the single policy/risk/planning authority for external capability descriptors.
